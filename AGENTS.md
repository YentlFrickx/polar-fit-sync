## STYLE

- Atomic file writes: always write to `{path}.fit.tmp` then `os.rename()` to final path; DB record inserted only after rename. Pattern used in `sync.py:_atomic_write`.
- APScheduler async jobs: register `async def` callables with `AsyncIOScheduler` directly â€” it awaits them and surfaces exceptions. Never wrap in a sync function calling `ensure_future` (swallows exceptions into orphan tasks).
- OAuth manual implementation: use raw `httpx` with `Authorization: Basic base64(client_id:client_secret)` header and form-body POST. No `authlib` needed for a single authorization-code exchange.
- FIT-byte parsing is isolated in one function (`sync.py:_parse_fit_sport`) so `fitparse` usage stays in one place, mirroring the existing convention of isolating third-party/IO concerns (`polar.py` for HTTP, `db.py` for SQL).
- Prefer recomputing a pure decision function (e.g. `_passes_filter(stored_value, current_config)`) over persisting a config signature/hash for cache-invalidation. A skip-tracking feature (`skipped_exercise` table, see ARCH_DECISIONS) originally proposed a `filter_signature` column + comparison scheme; an adversarial review (`OBJECTIONS_SKIP_TRACKING.md` O1) found it unnecessary because the underlying pure filter function can just be re-run against the stored value at read time â€” same correctness, one fewer column, no signature-staleness bugs. Reach for this pattern whenever "did config X change in a way that invalidates cached decision Y" comes up and Y was computed by a pure function.
- A resource's parent directory should be created by the module that owns/opens that resource, not by every caller. `Db.__init__` (`db.py`) now creates `pathlib.Path(path).parent` (guarding `":memory:"` and bare-filename/no-dir-component paths) instead of relying on callers to `mkdir` it first. This covers every entry point (`web.py`'s `create_app`, `__main__.py`'s `sync` command, and all tests) with one edit, and is why `PFS_DB_PATH` and `PFS_OUTPUT_DIR` (which still gets its own `mkdir` in `__main__.py`/`sync.py`, since no single module owns "the output dir") can safely point at totally unrelated directory trees.

## GOTCHAS

- `Dockerfile` editable install: `pip install -e .` with hatchling requires `src/` to be present first. Either use `pip install .` (non-editable) or `COPY src/ src/` before the pip install step. The original Dockerfile had this wrong and was fixed by the deploy agent.
- SQLite + NFS: never put a SQLite DB on an NFS-backed RWX PVC with multiple writers. The architecture was redesigned to use a single Deployment (one pod, one writer) with an RWO PVC to avoid this.
- `AsyncIOScheduler` + sync wrapper: a synchronous job calling `asyncio.get_event_loop()` is deprecated in Python 3.12+ and loses exceptions. Always register async job functions.
- Polar `expires_in`: the token exchange response includes `expires_in` even though long-lived. Store it and detect expiry by computation + 401 responses. Never assume tokens are permanent.
- `with_suffix(".tmp")` replaces the extension; `with_name(name + ".tmp")` appends. Use the latter for temp files so the original extension is preserved.
- Polar AccessLink's exercise-list `sport` field is coarser than the FIT file's own session `sport`: distinct activity types (e.g. walking vs. generic/uncategorized) can both report as `OTHER` at the API level. Never trust `ex.sport` for fine-grained classification â€” parse the downloaded FIT bytes (`fitparse`, first `session` message) when precision matters, and treat the API field as a fallback only.
- `docker-compose.yml` uses a named volume (`pfs-data`) for `/data`, not a host bind mount. The repo-root `data/` directory some local runs leave behind (e.g. from a bare `python -m polar_fit_sync` invocation outside Docker) is a separate, stale path â€” it is NOT what the running container reads or writes. When verifying file output against a Docker Compose deployment, always inspect the actual named volume (`docker run --rm -v polar-fit-sync_pfs-data:/data alpine ls /data/fit`), never the host-side `data/` directory.
- `tests/test_config.py::test_default_filter_empty` (and potentially sibling config tests) can fail spuriously if a local `.env` file sets `PFS_SPORT_FILTER`/other `PFS_*` vars â€” `config.py`'s `SettingsConfigDict(env_file=".env")` loads it and overrides the field default the test asserts on. Confirmed via `git stash`: this is pre-existing and unrelated to any specific change, not a regression. Tracked in `BACKLOG.md` as a test-hardening item.
- APScheduler `add_job(..., next_run_time=...)`: verified directly against the installed `apscheduler` source (`base.py`). Passing `next_run_time=None` does NOT mean "use the trigger's normal schedule" â€” the docstring says "pass `None` to add the job as paused," and a paused job's `next_run_time` is never set, so it **never fires again**. Omitting the kwarg entirely (the `undefined` sentinel default) is what makes APScheduler compute the trigger's normal first-fire time (`now + interval`). If you ever need to conditionally disable a "run now" acceleration, build the `add_job` kwargs dict and only add the `next_run_time` key in the enabled branch â€” never pass `next_run_time=None` thinking it's equivalent to "no override." (`scheduler.py:build_scheduler`, `PFS_SYNC_ON_STARTUP` feature.)
- Polar AccessLink's `GET /v3/exercises` list endpoint only ever returns exercises **uploaded to Flow in the last ~30 days** â€” verified against the authoritative OpenAPI/Swagger spec (`https://www.polar.com/accesslink-api/swagger.yaml`), not just a marketing page. The window is keyed on upload date, not activity date, and there is no date-range/pagination query param (only `samples`/`zones`/`route`), nor a full-history alternative endpoint (the deprecated `/v3/users/{id}/exercise-transactions` carries the same 30-day limit). Any client-side date filter (e.g. `PFS_SYNC_START_DATE`) can therefore only ever operate *within* that rolling ~30-day window â€” it cannot reach or exclude older historical data, because that data is never returned to us in the first place. Also: `start_time` is documented as *local time with no offset* (example `'2008-10-13T10:40:02'`, no `Z`/timezone suffix), with a separate `start_time_utc_offset` (minutes) field the codebase does not consume â€” any code parsing `start_time` with `fromisoformat` must handle the resulting naive datetime rather than assuming it's always tz-aware.

## ARCH_DECISIONS

- Single Deployment with in-process APScheduler instead of a separate K8s CronJob. Reason: two pods sharing a SQLite PVC risks concurrent-write corruption (especially on NFS). One pod, one writer, no coordination needed.
- Sync modes (`poll`, `webhook`, `both`) controlled by `PFS_SYNC_MODE` env var. Webhook mode requires `PFS_WEBHOOK_SECRET` and a public ingress; poll mode works with no public exposure.
- Token expiry handling: detect computed expiry (`now > created_at + expires_in`) at run start and 401 responses mid-run; flip `token.status` to `token_expired`; show re-link prompt on web UI. No token refresh (Polar does not support it).
- `POLAR_CLIENT_ID` / `POLAR_CLIENT_SECRET` required only for the `web` command. The `sync` command reads the stored access token from SQLite and does not need client credentials.
- `skipped_exercise` SQLite table (exercise_id PK, sport, skipped_at) remembers exercises filtered out post-download so they are not redundantly re-downloaded and re-parsed on every sync (`sync.py`'s work-list loop, `db.py`'s `record_skipped`/`list_skipped_sports`/`delete_skipped`). Staleness under a filter-config change is resolved by recomputing `_passes_filter(stored_sport, current_filter, current_mode)` at read time rather than persisting a filter signature â€” see STYLE entry above. `record_downloaded` and `delete_skipped` are two separate non-atomic `Db` calls (matching the codebase's one-connection-per-call convention); a crash between them can leave a stale `skipped_exercise` row alongside a `downloaded_exercise` row, which is harmless only because the work-list loop checks `is_downloaded` strictly first and short-circuits before ever consulting skip state for that id â€” this precedence must be preserved if the loop is touched again.

## DESIGN_DECISIONS

- `OAUTH_STATE_TTL_SECONDS = 600` (10 minutes) â€” defined in `db.py`, the single source. States older than 600s are rejected by `consume_state`.
- Dedup key for downloaded exercises = `exercise_id` (Polar's stable hashed id). `INSERT OR IGNORE` on `downloaded_exercise` table.
- Token table is a single-row table (`id=1` CHECK constraint). Only one Polar account supported.
- `sync_run.trigger` values: `poll` | `webhook` | `manual` | `startup`. `sync_run.status` values: `ok` | `partial` | `no_token` | `token_expired` | `error`.
- Sync exit code (manual CLI path): 0 for `ok`/`no_token`/`token_expired`; 1 for `partial`/`error`. This prevents the Kubernetes container from hard-failing on auth issues that require human re-link.
- Polar webhook signature: HMAC-SHA256 over the raw request body using `PFS_WEBHOOK_SECRET`; verified via `hmac.compare_digest` (constant-time). Header name: `Polar-Webhook-Signature`.
- Sport filtering: `PFS_SPORT_FILTER` (comma list) + `PFS_SPORT_FILTER_MODE` (`include`|`exclude`). Empty list = no filter. Matching is case-insensitive on uppercased sport name. `include` + null sport â†’ dropped; `exclude` + null sport â†’ kept. Applied in `run_sync` via `_passes_filter`; parsed once at call time by `Settings.sport_filter_set()`. Filter is forward-only â€” never deletes or backfills already-downloaded files. As of the FIT-parsed-sport fix, the value passed to `_passes_filter` is the sport parsed from the downloaded FIT file's `session` message (uppercased), not the coarse Polar API `sport` field â€” Polar's list API collapses walking and true-generic activities to the same string (`OTHER`), which is only resolvable by parsing the actual FIT bytes. Filter application moved from pre-download (on the exercise list) to per-exercise post-download (after parse, before write/record). Fallback to the coarse API sport happens on any parse failure (exception, missing session message, or missing sport field), logged as a warning, never counted as an error.
- Skip-tracking uses INSERT OR REPLACE (not IGNORE) for `record_skipped`, unlike `record_downloaded`'s INSERT OR IGNORE â€” a re-skip must refresh the stored sport and timestamp so a later recompute uses the latest FIT-parsed sport, whereas a download record is written once and never needs updating. The `list_skipped_sports()` lookup is batched (one query per run) rather than per-exercise like `is_downloaded`, specifically to avoid doubling per-item SQLite round trips on every poll cycle (flagged in `OBJECTIONS_SKIP_TRACKING.md` O6, accepted as a design constraint even though the objection itself was rejected).
- Startup sync acceleration: `PFS_SYNC_ON_STARTUP` (default `true`, poll/both only) makes the existing `polar_sync` interval job's *first* fire happen immediately instead of waiting a full `PFS_SYNC_INTERVAL_MINUTES`. Mechanism: `build_scheduler` (`scheduler.py`) conditionally passes `next_run_time=datetime.now(timezone.utc)` into the same `add_job(...)` call used today â€” there is no second job and no `asyncio.create_task`; the accelerated run is dispatched and tracked by APScheduler exactly like every other interval fire, so `scheduler.shutdown(wait=False)` already covers it. The distinct `sync_run.trigger="startup"` label (vs. `"poll"`) is produced by a first-run flag captured by the single shared `_sync_runner` closure in `web.py` (a one-element list cell, e.g. `_first_run_pending = [settings.pfs_sync_on_startup]`, flipped to `False` after its first use) â€” not by a second closure â€” so there is exactly one place that builds the `run_sync(...)` call args. When `PFS_SYNC_ON_STARTUP` is false, the cell starts `False` and the first run is labelled `poll` like every run before this feature existed. This design was chosen over an earlier `asyncio.create_task(_startup_sync())` draft specifically to reuse APScheduler's own exception surfacing/shutdown tracking instead of duplicating the runner closure â€” see `OBJECTIONS_SYNC_ON_STARTUP.md` O1/O5/O6.
- Sync start-date gating: `PFS_SYNC_START_DATE` (`YYYY-MM-DD`, empty = disabled) is parsed once per run by `Settings.sync_start_date()` into an aware UTC `datetime` at `00:00:00` on that day, then passed into `run_sync(start_date=...)` and applied per-exercise by the pure helper `_after_start_date(start_time, start_date)` in `sync.py`. Contract: (1) boundary is inclusive (`start_time >= start_date` is kept); (2) it is a pre-download gate â€” placed in the work-list loop immediately after the `is_downloaded` dedup check and before the skip-tracking block, so `is_downloaded` still takes strict precedence and a date-excluded exercise never writes a `skipped_exercise` row (that table is reserved for sport-filter skips only); (3) gates exclusively on activity `start_time`, never `upload_time` â€” a deliberate, accepted limitation (an old workout re-uploaded to Polar after the cutoff is set stays excluded if its activity date predates the cutoff), mirroring how the multisport FIT edge case is documented as accepted rather than silently gapped; (4) a null `start_time` is kept silently (no log â€” expected/benign, matching `_build_path`'s tolerance for missing dates); an unparseable `start_time` is also kept but logs at **WARNING** (not DEBUG) so a systemic Polar date-format drift stays visible under the shipped default `PFS_LOG_LEVEL=INFO`; (5) `start_time` can arrive as offset-less local time (see GOTCHAS), so the parsed value is normalized naiveâ†’UTC before comparison, following the same idiom already used for token-expiry parsing (`sync.py` token-expiry check), not `_build_path` (which never compares datetimes and so never needed this normalization); (6) forward-only â€” never deletes or reclassifies already-downloaded files; (7) config-only, no live-reload â€” changing the env var requires a full restart, consistent with every other `PFS_*` var (notably `PFS_SPORT_FILTER`).
